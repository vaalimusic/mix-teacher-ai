# ТЗ: Умный алгоритм анализа и подсказок для VST-плагина “Mix Teacher AI”

## 1. Цель алгоритма

Алгоритм должен в реальном времени анализировать аудио дорожки в DAW и давать понятные подсказки пользователю:

* что сейчас происходит со звуком;
* есть ли клиппинг;
* слишком тихо или слишком громко;
* что выбивается по частотам;
* есть ли муть, резкость, слабая атака, плохая динамика;
* насколько выводам можно доверять;
* что попробовать исправить;
* почему именно это действие может помочь.

Главная цель — не “магически свести трек”, а **объяснить пользователю проблему и безопасный первый шаг**.

---

# 2. Главная философия

Алгоритм не должен делать выводы по одному числу.

Плохо:

```text
5–9 kHz high → значит сибилянты → поставь de-esser
```

Хорошо:

```text
Track Type = Vocal
Сигнал достаточно громкий для анализа
В 5–9 kHz есть короткие резкие пики
Эти пики совпадают с транзиентами/согласными
Значит возможны сибилянты
Confidence: 74%
Совет: попробуй de-esser 6–8 kHz, reduction 2–4 dB
```

Алгоритм должен думать этапами:

```text
1. Можно ли вообще доверять анализу?
2. Что это за дорожка?
3. Какой у неё уровень?
4. Какие частоты реально доминируют?
5. Это проблема или нормальная особенность инструмента?
6. Насколько проблема важная?
7. Что посоветовать безопасно?
8. Как объяснить это простым языком?
```

# 4. Режимы анализа

## 4.1 Realtime Mode

Обновление каждые 100–250 ms.

Показывать:

* peak;
* RMS;
* waveform;
* spectrum;
* clipping;
* headroom;
* live frequency bands.

## 4.2 Smart Hint Mode

Обновление каждые 2–5 секунд.

Показывать:

* “похоже, есть муть”;
* “дорожка слишком тихая для уверенного анализа”;
* “пики близко к клиппингу”;
* “динамика сильно скачет”;
* “верх слишком резкий”.

## 4.3 Deep Analysis Mode

Пользователь нажимает кнопку:

```text
Analyze 15 sec
```

Плагин копит данные 10–30 секунд и выдаёт более точный отчёт.

## 4.4 AI Explanation Mode

Пользователь нажимает:

```text
Ask AI
```

Плагин отправляет структурированный JSON в локальную Ollama/AI и получает объяснение человеческим языком.

# 5. Первый обязательный этап: Validity Check

Перед любыми подсказками алгоритм должен понять, можно ли доверять анализу.

## 5.1 Проверка уровня сигнала

Если сигнал слишком тихий:

```text
peak < -35 dBFS
или
rms < -45 dBFS
```

Вывод:

```text
Сигнал слишком тихий для анализа.
Проиграй более громкий участок или подними уровень мониторинга.
```

Если сигнал тихий, но не тишина:

```text
peak < -18 dBFS
и
rms < -30 dBFS
```

Вывод:

```text
Дорожка очень тихая. Клиппинга нет, но частотные выводы могут быть неточными.
```

В этом случае запрещено давать уверенные советы типа:

```text
убери муть
поставь de-esser
компрессируй
```

Пока можно советовать только:

```text
проверь уровень;
проиграй более насыщенный участок;
выбери тип дорожки вручную.
```

# 6. Active Signal Detection

Алгоритм должен отличать музыку от пауз.

## 6.1 Не учитывать тишину в динамике

Нельзя считать `rms_range` по всему фрагменту, если там есть паузы.

Нужно:

```text
1. Разбить сигнал на окна по 100–200 ms.
2. Посчитать RMS каждого окна.
3. Найти активные окна.
4. Исключить тишину и хвосты.
5. Считать динамику только по активным окнам.
```

Пример:

```text
active_window = rms > max_rms - 30 dB
```

Или:

```text
active_window = rms > -50 dBFS
```

## 6.2 Percentile Range вместо полного range

Плохо:

```text
max RMS - min RMS
```

Хорошо:

```text
P90 RMS - P10 RMS
```

Так паузы не ломают анализ.

# 7. Определение типа дорожки

## 7.1 Ручной режим

В первой версии пользователь выбирает:

```text
Vocal
Kick
Snare
Bass
Drums
Guitar
Piano
Synth
FX
Master
Auto
```

Ручной выбор должен иметь приоритет над Auto.

## 7.2 Auto Detect

Auto Detect должен выдавать:

```json
{
  "detected_track_type": "drums",
  "confidence": 0.72,
  "alternatives": [
    {"type": "percussion_loop", "confidence": 0.61},
    {"type": "master_loop", "confidence": 0.44}
  ]
}
```

## 7.3 Пример признаков для классификации

### Vocal

Признаки:

* много энергии 100 Hz – 8 kHz;
* нет постоянного сильного саба;
* есть формантная структура;
* есть сибилянтные пики 5–9 kHz;
* динамика фразовая;
* transient score средний.

### Kick

Признаки:

* сильный пик 40–100 Hz;
* короткие транзиенты;
* повторяющиеся удары;
* мало информации в середине, кроме атаки.

### Bass

Признаки:

* устойчивая энергия 40–250 Hz;
* меньше транзиентов, чем у кика;
* есть ноты;
* может быть атака 700 Hz – 2 kHz.

### Drums

Признаки:

* много транзиентов;
* широкий спектр;
* активны низ, середина и верх;
* повторяющийся ритмический паттерн.

### Guitar

Признаки:

* основная энергия 100 Hz – 5 kHz;
* мало саба;
* атака медиатора 2–5 kHz;
* возможная резкость 3–6 kHz.

# 8. Feature Extraction

Алгоритм должен собирать не только готовые ярлыки, а реальные численные признаки.

## 8.1 Levels

```json
{
  "peak_dbfs": -6.2,
  "true_peak_dbfs": -5.7,
  "rms_dbfs": -18.4,
  "lufs_short_term": -17.9,
  "crest_factor_db": 12.2,
  "headroom_db": 6.2,
  "clipping_count": 0,
  "clipping_detected": false
}
```

## 8.2 Dynamics

```json
{
  "active_rms_p10": -28.1,
  "active_rms_p50": -20.4,
  "active_rms_p90": -16.2,
  "active_rms_range_db": 11.9,
  "transient_score": 0.64,
  "onset_count": 42,
  "peak_to_rms_db": 12.2
}
```

## 8.3 Spectrum

Нужно хранить реальные значения в dB, а не только `high/medium/low`.

```json
{
  "band_energy_db": {
    "20_40_hz": -42.3,
    "40_80_hz": -28.1,
    "80_150_hz": -25.4,
    "150_350_hz": -21.8,
    "350_800_hz": -29.2,
    "800_2khz": -27.5,
    "2_5khz": -24.1,
    "5_9khz": -22.9,
    "9_16khz": -34.7
  }
}
```

Потом отдельно вычислять:

```json
{
  "dominant_bands": [
    "150_350_hz",
    "5_9khz",
    "2_5khz"
  ],
  "weak_bands": [
    "9_16khz"
  ]
}
```

# 9. Относительный анализ вместо грубого high/low

Проблема старого подхода:

```json
"20_40_hz": "high",
"40_80_hz": "high",
"80_150_hz": "high",
"180_350_hz": "high"
```

Если всё high — это не анализ.

Нужно оценивать полосы относительно:

```text
1. общего уровня дорожки;
2. соседних полос;
3. ожидаемого профиля инструмента;
4. жанрового профиля;
5. master/reference, если доступно.
```

## 9.1 Пример

Для вокала `180–350 Hz` считается проблемной зоной только если:

```text
Track Type = Vocal
Signal Validity = Good
Energy 180–350 Hz выше соседних зон
Energy 180–350 Hz выше вокального профиля
Presence 2–5 kHz не компенсирует мутность
Confidence > 0.6
```

# 12. Правила по типам дорожек

## 12.1 Vocal

Возможные проблемы:

```text
too_hot
too_quiet
mud
boxiness
sibilance
harshness
weak_presence
dullness
unstable_dynamics
overcompressed
```

Запрещённые или осторожные советы:

```text
Не советовать de-esser, если track_type не Vocal.
Не советовать сильный boost 10–16 kHz, если уже есть sibilance.
Не советовать компрессию, если сигнал слишком тихий для анализа.
```

## 12.2 Kick

Возможные проблемы:

```text
sub_rumble
weak_fundamental
too_much_100_150
boxiness
weak_click
too_spiky
long_tail
```

Советы:

```text
HPF 25–35 Hz
body 50–80 Hz
cut 100–150 Hz только если конфликт/бубнёж
cut 200–500 Hz для коробочности
click 2–5 kHz
```

## 12.3 Bass

Возможные проблемы:

```text
sub_too_strong
weak_note_definition
too_much_low_mid
uneven_notes
too_dynamic
too_compressed
```

Советы:

```text
контроль 40–120 Hz
добавить читаемость 700 Hz – 1.5 kHz
мягкая компрессия
sidechain от kick только в контексте микса
```

## 12.4 Drums / Drum Bus

Возможные проблемы:

```text
too_quiet
too_hot
weak_transients
overcompressed
boxy_drums
harsh_cymbals
low_end_buildup
clipping
```

Важно:

```text
Не использовать слово "sibilance" для барабанов.
Для 5–9 kHz писать "резкие хэты/тарелки/атака", а не "сибилянты".
```

## 12.5 Master

Возможные проблемы:

```text
master_clipping
low_headroom
too_loud_before_mastering
too_much_low_end
too_bright
weak_mids
overcompressed_mix
```

Советы:

```text
до мастеринга target peak примерно -10 до -6 dBFS
не лечить баланс лимитером
проверить группы/дорожки
```

# 13. Приоритеты подсказок

Алгоритм не должен показывать 10 проблем сразу.

Нужно выбрать максимум 3 главные проблемы.

## 13.1 Priority Order

```text
1. Clipping
2. Signal too quiet for analysis
3. Master/track too hot
4. Wrong/unknown track type
5. Dynamics problems
6. Major frequency problems
7. Minor tonal suggestions
```

## 13.2 Пример

Если есть клиппинг, не надо сразу писать про муть.

Сначала:

```text
Есть клиппинг. Сначала убери перегруз, потом анализ частот станет точнее.
```

Если сигнал слишком тихий:

```text
Сигнал слишком тихий. Частотные подсказки пока ненадёжны.
```

---

# 14. Confidence System

Каждая проблема должна иметь confidence:

```text
0.0 – 0.3 = low confidence
0.3 – 0.6 = possible
0.6 – 0.8 = likely
0.8 – 1.0 = strong
```

UI должен показывать это простыми словами:

```text
Возможно
Похоже
Очень похоже
Почти точно
```

Пример:

```text
Похоже, есть муть в районе 220–300 Hz.
Уверенность: 78%
```

# 15. Explanation Engine

Каждая подсказка должна иметь 4 части:

```text
1. Что не так
2. Почему это может мешать
3. Что попробовать
4. Как проверить на слух
```

## Пример

```text
Что не так:
Кик громкий по низу, но плохо читается.

Почему:
У него много энергии в 50–100 Hz, но мало атаки в 2–5 kHz. На больших колонках он есть, а на маленьких может пропадать.

Что попробовать:
Не поднимай громкость. Лучше попробуй добавить 2–3 dB в районе 3–4 kHz или убрать немного 250–400 Hz.

Как проверить:
Сделай тише мониторинг и послушай, слышен ли ритм кика без саба.
```

# 16. Beginner / Intermediate / Advanced

## 16.1 Beginner

```text
Дорожка слишком громкая.
Убавь input gain на 3–5 dB.
```

## 16.2 Intermediate

```text
Пики близко к 0 dBFS, headroom всего 1.4 dB.
Лучше убавить уровень до обработки, а не только фейдером.
```

## 16.3 Advanced

```text
Peak: -1.4 dBFS, RMS: -14.2 dBFS, crest factor: 7.1 dB.
Сигнал плотный и близок к цифровому потолку. Рекомендуется снизить clip/input gain на 3–5 dB перед insert-chain.
```

# 17. “Вау”-функции

## 17.1 Mix Buddy

В интерфейсе есть персонаж-помощник.

Состояния:

```text
🙂 Всё спокойно
😬 Осторожно, близко к клиппингу
😵 Клиппинг
🤔 Я не уверен, что это за дорожка
🧼 Похоже, есть муть
🔪 Верх режет ухо
🎚️ Убавь входной уровень
🥁 Это похоже на барабаны
🎤 Это похоже на вокал
```

## 17.2 “Почему кажется громко?”

Плагин должен объяснять, почему дорожка воспринимается громкой:

```json
{
  "perceived_loudness_reasons": [
    "high RMS",
    "strong 2-5 kHz presence",
    "low crest factor",
    "peaks close to clipping"
  ]
}
```

UI:

```text
Дорожка кажется громкой не только из-за фейдера.
Причины:
- плотный RMS;
- мало динамики;
- активная зона 2–5 kHz.
```

## 17.3 “Сначала исправь это”

Плагин должен выбирать один главный шаг:

```text
Главный первый шаг:
Убавь clip/input gain на 4 dB.
```

А не заваливать пользователя списком.

## 17.4 “До / После”

Кнопка:

```text
Freeze Before
```

Пользователь делает правку, потом:

```text
Compare After
```

Плагин показывает:

```text
До:
Peak -1.2 dBFS, mud high

После:
Peak -6.4 dBFS, mud medium

Стало лучше:
- больше headroom;
- меньше мути;
- клиппинга нет.
```

## 17.5 “Учебный режим”

Плагин объясняет термины:

```text
Что такое headroom?
Это запас до 0 dBFS. Если запаса мало, плагины и мастер могут начать клипповать.
```

# 19. Правила безопасности

Алгоритм не должен советовать экстремальные настройки.

Ограничения:

```text
EQ cut: обычно -1 до -6 dB
EQ boost: обычно +1 до +4 dB
HPF для вокала: 60–120 Hz
HPF для кика: 20–40 Hz
HPF для баса: 20–40 Hz
Compressor gain reduction: 2–6 dB
De-esser reduction: 2–5 dB
Input gain correction: -2 до -12 dB
```

Если нужна сильная правка, писать:

```text
Если нужно больше -6 dB, возможно проблема в выборе сэмпла, записи или arrangement.
```

# 20. Mix Hub v0.3

Для настоящего “вау” нужна система из нескольких экземпляров.

## 20.1 Track Plugin

Стоит на каждой дорожке и отправляет данные:

```text
Kick
Bass
Vocal
Snare
Drums
Guitar
Master
```

## 20.2 Mix Hub

Стоит на master bus и собирает данные от всех экземпляров.

## 20.3 Что сможет Mix Hub

```text
Kick vs Bass conflict
Vocal vs Instrumental balance
Drums vs Bass balance
Master headroom
Cymbals too bright against vocal
Bass too loud against kick
Vocal too quiet in chorus
```

## 20.4 Пример умной подсказки

```text
Кик и бас конфликтуют в 80–110 Hz.

Почему:
У кика и баса одновременно много энергии в одной зоне. На мастере это может давать гул и перегруз.

Что попробовать:
- решить, кто главный в 60–90 Hz;
- если кик главный, немного убрать эту зону у баса;
- если бас главный, добавить кику атаку 2–4 kHz;
- можно попробовать sidechain 2–4 dB от кика к басу.
```

# 21. Главные критерии качества алгоритма

Алгоритм считается хорошим, если он:

```text
1. Не даёт вокальные советы барабанам.
2. Не делает частотные выводы, если сигнал слишком тихий.
3. Не считает паузы проблемой динамики.
4. Не пишет “всё high”.
5. Показывает confidence.
6. Выбирает 1–3 главные проблемы, а не 15.
7. Даёт безопасный первый шаг.
8. Объясняет почему.
9. Учит пользователя слушать, а не только смотреть на цифры.
10. Работает без AI, но может улучшаться через Ollama.
```

# 23. Главная формула поведения

Алгоритм должен работать по принципу:

```text
Сначала проверить уровень.
Потом определить тип дорожки.
Потом понять, можно ли доверять анализу.
Потом найти 1–3 главные проблемы.
Потом объяснить простым языком.
Потом дать безопасный первый шаг.
Потом предложить проверить на слух.
```

Не так:

```text
Частота высокая → проблема → совет.
```

А так:

```text
Контекст → измерение → уверенность → проблема → объяснение → безопасное действие → проверка слухом.
```
